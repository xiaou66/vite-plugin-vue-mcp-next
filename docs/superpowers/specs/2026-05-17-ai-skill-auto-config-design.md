# AI 使用指南 Skill 自动配置设计

## 背景

`vite-plugin-vue-mcp-next` 已经能在 Vite dev server 中暴露 MCP（Model Context Protocol，模型上下文协议）服务，并自动写入常见 AI 客户端的 MCP 连接配置。但“连接上 MCP”不等于 AI 会正确使用这些工具。不同 AI 客户端需要一份项目级指导，告诉它在 Vue/Vite 调试场景里什么时候使用本 MCP、先调用哪个工具、哪些工具有副作用，以及 CDP、Runtime Hook、Vue Runtime Bridge 的边界。

这次新增的是项目启动时自动安装 AI 使用指南。它不改变 MCP server 本身，只把一份可复用的指导内容写入各客户端支持的项目级 skill 或 rule 目录。

## 术语说明

| 术语 | 说明 |
| --- | --- |
| MCP（Model Context Protocol，模型上下文协议） | AI 客户端连接本地工具服务的协议，本插件通过它暴露 Vue/Vite 调试能力 |
| Skill | AI 客户端可按需加载的项目级能力说明，通常由 `SKILL.md` 承载 |
| Rule | AI 客户端的持久项目规则，用于提供稳定上下文和工具使用约束 |
| 自动安装 | Vite dev server 启动时，插件按项目中已有客户端入口写入 skill 或 rule 文件 |
| 高风险工具 | 会读取或修改页面运行时状态的工具，例如 `evaluate_script` 和 `edit_component_state` |

## 已确认事实

- Claude Code 官方文档说明项目级 skill 路径是 `.claude/skills/<skill-name>/SKILL.md`，并且 `SKILL.md` 是 skill 入口文件。
- Cursor 官方文档说明项目规则存放在 `.cursor/rules`，每个规则文件使用 `.mdc` 格式；Cursor 这里应使用 Rule，不应伪装成 Skill。
- Codex 当前运行环境已支持项目级 `.codex/skills/<skill-name>/SKILL.md`，本仓库也已有 `.codex/skills/` 形态可被 Codex 识别。
- Trae 已有 `.trae/mcp.json` MCP 配置入口；其 rules 路径公开资料不如 Claude Code 和 Cursor 稳定，因此首版不自动写入 Trae rule，只在 README 说明手动配置边界。

## 目标

- 新增顶层配置 `skill.autoConfig`，默认 `true`
- 用户设置 `skill.autoConfig: false` 后，启动 Vite 时不写任何 skill 或 rule 文件
- 在项目启动时自动为已检测到的客户端安装 `vite-mcp-next` 使用指南
- Skill/Rule 内容要明确 AI 使用本 MCP 的场景、工具顺序、风险边界和回退策略
- 复用现有自动配置理念：默认只处理项目中已经存在的客户端入口，避免污染未使用的客户端目录
- README、类型注释和测试同步说明新配置

## 非目标

- 不改变 MCP transport、SSE、Streamable HTTP、Runtime Hook、CDP 或 Vue Runtime Bridge 实现
- 不新增 MCP 工具
- 不自动启动浏览器或 CDP 端口
- 不覆盖用户已经手写且内容不同的 skill/rule 文件
- 不把 Cursor 或 Trae 的规则文件描述为标准 Agent Skill
- 不依赖网络在运行时拉取模板，模板内容必须随 npm 包一起发布

## 配置设计

新增顶层配置：

```ts
vueMcpNext({
  skill: {
    autoConfig: true
  }
})
```

`skill.autoConfig` 的语义：

| 配置值 | 行为 |
| --- | --- |
| `undefined` | 使用默认值 `true`，按客户端入口自动安装 |
| `true` | 按客户端入口自动安装 |
| `false` | 完全跳过 skill/rule 自动安装 |

首版不增加按客户端拆分的 skill 开关。原因是 MCP 客户端配置已经承担“是否接入某客户端”的职责，而 skill 自动安装只是给已接入或已存在入口的客户端补充使用指导。后续如果用户需要精细控制，再扩展为：

```ts
vueMcpNext({
  skill: {
    autoConfig: true,
    clients: {
      codex: true,
      claudeCode: false
    }
  }
})
```

## 自动安装路径

| 客户端 | 探测入口 | 写入路径 | 文件类型 | 首版行为 |
| --- | --- | --- | --- | --- |
| Codex | `.codex/` | `.codex/skills/vite-mcp-next/SKILL.md` | Skill | 自动写入 |
| Claude Code | `.claude/` | `.claude/skills/vite-mcp-next/SKILL.md` | Skill | 自动写入 |
| Cursor | `.cursor/` | `.cursor/rules/vite-mcp-next.mdc` | Rule | 自动写入 |
| Trae | `.trae/` | 不写入 | Rule | 首版跳过自动安装，只在 README 说明手动配置边界 |

默认自动安装只在探测入口存在时执行。例如项目没有 `.claude/` 时，不主动创建 `.claude/skills/`。这样可以避免普通 Vite 项目启动后出现未使用客户端的配置目录。

## Skill/Rule 内容设计

统一模板名称为 `vite-mcp-next`。不同客户端只调整 frontmatter 或规则头部格式，正文保持同一份语义。

内容必须覆盖：

1. 适用场景
   - Vue/Vite 本地页面调试
   - DOM 结构、页面截图、Console 日志、Network 请求排查
   - Vue component、Router、Pinia 状态检查
   - 用户要求“用 MCP 看页面”“截图验证”“查接口请求”“检查组件状态”时优先使用

2. 推荐工具顺序
   - 先调用 `list_pages` 确认页面、runtime 和 CDP target
   - 页面结构问题用 `get_dom_tree` 或 `query_dom`
   - 视觉验证用 `take_screenshot`
   - 报错排查用 `get_console_logs`
   - 接口排查用 `get_network_requests` 和 `get_network_request_detail`
   - Vue 语义问题用 `get_component_tree`、`get_component_state`、`get_router_info`、`get_pinia_tree`、`get_pinia_state`

3. 通道边界
   - DOM、Console、Network、Screenshot 在 CDP 可用时优先使用 CDP
   - CDP 不可用时走 Runtime Hook 或 snapdom 降级
   - Vue component、Router、Pinia 固定走 Vue Runtime Bridge，不用 CDP 替代

4. 风险边界
   - `evaluate_script` 默认关闭，只有 `runtime.evaluate.enabled: true` 时才可用
   - `evaluate_script` 应优先用于只读诊断，不默认修改页面状态
   - `edit_component_state` 会修改运行时组件状态，必须确认用户目标需要修改
   - Network 数据可能包含敏感内容，读取后不要把 token、cookie、authorization 明文写入代码或文档

5. 失败处理
   - `list_pages` 没有 runtime 页面时，先提示启动或刷新 Vite 页面
   - CDP target 不存在时，回退到 runtime 页面能力
   - 截图失败时说明来源是 CDP 失败还是 snapdom 失败
   - MCP 未连接时，提示核对启动日志中的 `__mcp/sse` 或 `__mcp/mcp` 地址

## 文件写入规则

- 只写本插件拥有的目标文件
- 如果目标文件不存在，创建目录并写入模板
- 如果目标文件存在且包含插件生成标记，允许更新到当前模板版本
- 如果目标文件存在但没有插件生成标记，视为用户自定义文件，跳过并输出 `console.warn`
- 写入失败只警告，不阻断 Vite dev server 启动
- 生成文件顶部加入中文说明，解释文件用途和自动生成边界

建议生成标记：

```md
<!-- Generated by vite-plugin-vue-mcp-next. Safe to edit, but automatic updates only apply while this marker remains. -->
```

## 架构设计

新增独立模块 `src/plugin/skillConfig/`，避免继续膨胀 MCP client config 模块。

建议职责拆分：

| 模块 | 职责 |
| --- | --- |
| `skillConfig/index.ts` | 编排客户端探测和写入任务 |
| `skillConfig/template.ts` | 生成统一 Skill/Rule 正文 |
| `skillConfig/writers.ts` | 提供 `writeGeneratedTextFile()` 等安全写入能力 |
| `types.ts` | 暴露 `SkillConfigOptions` 和 resolved 类型 |
| `constants.ts` | 合并 `skill.autoConfig` 默认值 |

`configureServer()` 中现有流程增加一步：

1. 设置 MCP transport
2. 计算 MCP URL
3. 更新 MCP client configs
4. 更新 AI skill/rule configs
5. 打印 MCP URL

## 数据流

1. 用户在 `vite.config.ts` 中调用 `vueMcpNext(userOptions)`
2. `mergeOptions()` 合并 `skill.autoConfig` 默认值
3. `configureServer()` 计算项目根目录
4. 如果 `options.skill.autoConfig` 为 `false`，直接跳过
5. 如果开启，检测 `.codex/`、`.claude/`、`.cursor/`
6. 对已确认支持的客户端生成对应文件
7. 每个客户端独立捕获错误并输出警告

## 测试计划

新增或调整单元测试：

1. 默认配置下 `skill.autoConfig` 为 `true`
2. 用户配置 `skill.autoConfig: false` 后 resolved 配置保持关闭
3. 存在 `.codex/` 时写入 `.codex/skills/vite-mcp-next/SKILL.md`
4. 存在 `.claude/` 时写入 `.claude/skills/vite-mcp-next/SKILL.md`
5. 存在 `.cursor/` 时写入 `.cursor/rules/vite-mcp-next.mdc`
6. 目标文件存在且带生成标记时允许更新
7. 目标文件存在但不带生成标记时跳过并保留用户内容
8. `skill.autoConfig: false` 时，即使入口存在也不写入
9. 单个客户端写入失败不影响其他客户端

## 验收标准

- 默认启动时，已存在 `.codex/` 的项目自动得到 `.codex/skills/vite-mcp-next/SKILL.md`
- 默认启动时，已存在 `.claude/` 的项目自动得到 `.claude/skills/vite-mcp-next/SKILL.md`
- 默认启动时，已存在 `.cursor/` 的项目自动得到 `.cursor/rules/vite-mcp-next.mdc`
- `skill.autoConfig: false` 时不创建或修改任何 skill/rule 文件
- 已有用户自定义文件不会被覆盖
- 生成内容明确指导 AI 使用 `list_pages`、DOM、Screenshot、Console、Network、Vue、Router、Pinia 相关工具
- 生成内容明确标注 `evaluate_script` 和 `edit_component_state` 的风险边界
- README、类型注释和测试覆盖配置行为
- `pnpm run check` 和 `git diff --check` 通过
